Banking Data Pipeline

Author: Abigail Mwanza
Project: Banking Data Engineering Pipeline

What This Project Does

Imagine you run a bank. Every second, customers deposit money, withdraw cash, and transfer funds. All of this is recorded in a database. The question is: how do you get this information out of that database fast enough to catch fraud, run reports, and answer questions in real time?

This project is the answer. It is a pipeline — a chain of tools that carries data from the bank's main database all the way to a data warehouse, where analysts can ask questions with SQL.

The pipeline has 5 main stages: Generate → Capture → Stream → Land → Load. Each Python script in your project handles one of these stages.

The journey of one transaction

When a customer withdraws ZMW 500, here is what happens in your pipeline:

  1. Generate: The Python data generator pretends to be a customer and writes a WITHDRAWAL row into PostgreSQL.
  2. Capture: Debezium sees the new row in Postgres's internal log and shouts "new transaction!" to Kafka.
  3. Stream: Kafka holds the message and lets any program that's listening grab it.
  4. Land: Your consumer script listens to Kafka, takes each message, and saves it as a small file in MinIO (your file storage).
  5. Load: Airflow runs every minute. It grabs all the new files from MinIO and loads them into Snowflake, where anyone can run SQL queries.
  6. Transform: dbt then cleans up the raw data and builds nice summary tables for reporting.

Script 1 — The Data Generator

data-generator/faker_generator.py

What it does

This script pretends to be a busy Zambian bank. Every 2 seconds, it creates fake customers, fake accounts, and fake transactions, and writes them into the PostgreSQL database. This gives the rest of the pipeline a constant stream of data to work with — no real bank needed.

Analogy: Imagine a teller at a bank window who never goes home. Every 2 seconds, they serve 10 new customers, open 20 accounts, and process 50 transactions. That's what this script does — non-stop.

How it works, step by step

1. Setup — reading settings

load_dotenv()

Reads your .env file to get the Postgres password, username, and address. This keeps secrets out of the code.

2. Defining the "shape" of the data

NUM_CUSTOMERS = 10
ACCOUNTS_PER_CUSTOMER = 2
NUM_TRANSACTIONS = 50
CURRENCY = "ZMW"
MIN_TXN_AMOUNT = 10.00
MAX_TXN_AMOUNT = 10000.00

Each time the script runs, it makes 10 new customers, 2 accounts each (so 20 accounts), and 50 transactions. Amounts are in Zambian Kwacha, between ZMW 10 (airtime) and ZMW 10,000 (rent or salary).

3. Zambian names, not Western ones

ZM_FIRST_NAMES = ["Chanda", "Mulenga", "Bwalya", "Mwape", ...]
ZM_LAST_NAMES = ["Banda", "Phiri", "Tembo", "Zulu", ...]

The default Faker library gives names like "John Smith." You replaced it with real Zambian names from Bemba, Nyanja, Tonga, Lozi, and Tumbuka language groups. The emails also use Zambian providers like zamtel.co.zm and unza.zm.

4. Connecting to PostgreSQL

conn = psycopg2.connect(host=..., port=..., dbname=..., user=..., password=...)

Opens a connection to your banking database running in Docker.

5. The main loop — what run_iteration() does

For every round, the script:

  1. Creates 10 customers: picks a random Zambian first name, last name, and email, then inserts them into the customers table.
  2. Creates 2 accounts per customer: each account is randomly SAVINGS or CHECKING, with a starting balance between ZMW 100 and ZMW 25,000.
  3. Creates 50 transactions: each transaction is a random DEPOSIT, WITHDRAWAL, or TRANSFER between the accounts it just made.

6. The endless loop

while True:
    run_iteration()
    time.sleep(SLEEP_SECONDS)   # wait 2 seconds, then do it again

The script never stops on its own. You stop it with Ctrl+C. If you pass --once as a command-line flag, it runs a single round and exits.

Why this matters: Without this script, your Kafka, MinIO, Snowflake, and Airflow stages would have nothing to do. This is the heartbeat of your pipeline.

Script 2 — The Debezium Connector Setup

kafka-debezium/generate_and_post_connector.py

What it does

This script runs only once. Its job is to tell Debezium: "Please watch the banking database and send every change to Kafka."

Analogy: Debezium is like a security camera, and this script is how you install it. Once installed, the camera watches the database day and night without you having to touch it again.

Key concepts

What is Change Data Capture (CDC)?

Instead of asking Postgres every few seconds "anything new?" (which is slow and misses things), CDC reads Postgres's internal log — called the Write-Ahead Log (WAL) — where Postgres records every change before it commits. Debezium reads this log and republishes every change as a message on Kafka.

What is a Kafka Connect connector?

A "connector" is just a configuration — a JSON document — that tells Debezium:

  • Which database to watch (host, port, username, password)
  • Which replication slot to use (a bookmark in the WAL)
  • Which Kafka topics to post to
  • How to format the messages

How this script works

1. Build the config dictionary

connector_config = {
  "name": "banking-postgres-connector",
  "config": {
    "connector.class": "io.debezium.connector.postgresql.PostgresConnector",
    "database.hostname": "postgres",
    "database.port": "5432",
    ...
    "plugin.name": "pgoutput",
    "slot.name": "banking_slot",
    "topic.prefix": "banking",
    "decimal.handling.mode": "double"
  }
}

2. Key settings explained in simple terms

SettingSimple meaning
database.hostnameWhere to find Postgres. Inside Docker, this is postgres, not localhost.
plugin.name = pgoutputBuilt-in Postgres tool for reading the WAL. No extra install needed.
slot.name = banking_slotA bookmark. If Debezium restarts, it resumes from where it stopped — no data is lost.
topic.prefix = bankingEvery Kafka topic starts with "banking". So the customers table becomes topic banking.public.customers.
decimal.handling.mode = doubleMoney amounts arrive as normal numbers (1234.56), not as hard-to-read bytes.

3. Send the config to Debezium via HTTP

response = requests.post(
    "http://localhost:8083/connectors",
    headers={"Content-Type": "application/json"},
    data=json.dumps(connector_config)
)

Debezium exposes a web API. This script sends a POST request — the same way a form on a website is submitted — to register the connector.

4. Check the response

The script then checks if it worked:

  • 201 Created → connector is set up for the first time ✅
  • 200 OK → connector already existed and was updated ✅
  • 409 Conflict → connector already exists, delete it first if you want to recreate
  • 5xx → Debezium or Postgres is down
Important: After running this script once, Debezium starts watching Postgres forever. You only run this script again if you need to delete and rebuild the connector.

Script 3 — The Kafka to MinIO Consumer

consumer/kafka_to_minio.py

What it does

This script listens to Kafka forever. Every time a new banking change arrives, it takes the message, saves it as a small Parquet file, and uploads that file to MinIO (your cheap S3-style storage).

Analogy: Think of Kafka as a busy post office and this script as a receptionist with a filing cabinet. Every letter that arrives, she opens it, puts it in the correct folder (labeled by table and date), and files it neatly.

Why save to MinIO instead of directly into Snowflake?

  • Cheap: MinIO storage costs nothing; Snowflake charges you per query and per byte.
  • Replayable: if something breaks downstream, the raw events are still sitting in MinIO.
  • Batch-friendly: Snowflake loads files much faster than single-row inserts.

How it works, step by step

1. Subscribe to Kafka topics

consumer = KafkaConsumer(
    'banking.public.customers',
    'banking.public.accounts',
    'banking.public.transactions',
    bootstrap_servers=os.getenv("KAFKA_BOOTSTRAP"),
    auto_offset_reset='earliest',
    enable_auto_commit=True,
    group_id=os.getenv("KAFKA_GROUP"),
    value_deserializer=lambda x: json.loads(x.decode('utf-8'))
)

The key settings:

SettingSimple meaning
auto_offset_reset='earliest'First time running? Read from the very beginning, don't miss anything.
enable_auto_commit=TrueKafka remembers the last message you read. Restart safely — no duplicates.
group_idA name so Kafka tracks your progress separately from other consumers.
value_deserializerKafka delivers raw bytes; this converts them into a Python dictionary.

2. Connect to MinIO

s3 = boto3.client('s3',
    endpoint_url=os.getenv("MINIO_ENDPOINT"),
    aws_access_key_id=os.getenv("MINIO_ACCESS_KEY"),
    aws_secret_access_key=os.getenv("MINIO_SECRET_KEY"),
    region_name='us-east-1')

MinIO speaks the same language as Amazon S3, so you use Amazon's boto3 library. The region is fake — MinIO doesn't care, but boto3 insists.

3. Make sure the bucket exists

if bucket not in [b['Name'] for b in s3.list_buckets()['Buckets']]:
    s3.create_bucket(Bucket=bucket)

On the very first run, this creates the "raw" bucket (folder) inside MinIO.

4. The writing function

def write_single_to_minio(table_name, record):
    df = pd.DataFrame([record])
    date_str  = datetime.now().strftime('%Y-%m-%d')
    file_path = f'{table_name}_{datetime.now().strftime("%H%M%S%f")}.parquet'
    df.to_parquet(file_path, engine='fastparquet', index=False)
    s3_key = f'{table_name}/date={date_str}/{file_path}'
    s3.upload_file(file_path, bucket, s3_key)
    os.remove(file_path)

In plain English:

  1. Put the single row into a tiny table (DataFrame).
  2. Save it as a Parquet file on the local disk, with a unique filename using microseconds so files don't overwrite each other.
  3. Upload that file to MinIO, organized by table and date: transactions/date=2026-04-24/transactions_093012567834.parquet.
  4. Delete the local file — MinIO now has the only copy.

5. The endless loop

for message in consumer:
    topic   = message.topic
    event   = message.value
    payload = event.get("payload", {})
    record  = payload.get("after")

    if record:
        write_single_to_minio(topic.split('.')[-1], record)

Every Debezium message looks like this:

{
  "payload": {
    "before": { old row },
    "after":  { new row },
    "op": "c"   // c=insert, u=update, d=delete
  }
}

The script only looks at after (the current state of the row). If after is empty (a delete), it skips the message.

Script 4 — The Airflow DAG (MinIO to Snowflake)

docker/dags/minio_to_snowflake_dag.py

What it does

This is an Airflow DAG (a scheduled workflow). Every minute, Airflow runs it. The DAG has two tasks:

  1. download_minio: grab all Parquet files sitting in MinIO.
  2. load_snowflake: push them into Snowflake tables.
Analogy: Airflow is like a factory supervisor with a clipboard. Every minute, the supervisor says "OK, truck driver (task 1), go fetch the packages. OK, warehouse worker (task 2), put them on the right shelves." If anyone fails, the supervisor writes it down and sends the boss an email.

How it works

1. Load configuration

load_dotenv(
    dotenv_path=os.path.join(os.path.dirname(__file__), ".env"),
    override=True,
)

Reads the DAG's own .env file (the one in docker/dags/). The override=True flag makes sure this file wins over any environment variables set by Docker.

2. Snowflake settings

SNOWFLAKE_SCHEMA = "raw"   # hardcoded — this DAG lands raw data

The schema is fixed to raw because this DAG's only job is to land CDC data into the raw layer. (dbt will transform it later, writing to analytics.)

3. The failure callback

def failure_email_callback(context):
    ti = context.get("task_instance")
    ...
    send_email(to=ALERT_RECIPIENTS,
               subject=f"🚨 Banking Pipeline Failed — {dag_id}.{task_id}",
               html_content=...)

When a task fails, Airflow calls this function. It builds a detailed HTML email with:

  • The DAG name, task name, when it failed, and which attempt number
  • The actual Python error that broke it
  • The last 25 lines of the task's log file
  • A clickable link back to the full log in Airflow

Then it emails you at abigailmwnz@gmail.com.

4. Task 1 — download_from_minio

s3 = boto3.client('s3', ...)
for table in ["customers", "accounts", "transactions"]:
    resp = s3.list_objects_v2(Bucket=BUCKET, Prefix=f"{table}/")
    for obj in resp.get("Contents", []):
        s3.download_file(BUCKET, obj["Key"], local_file)

For each of the three tables, list every file in the MinIO folder, download it into /tmp/minio_downloads/ inside the container, and remember where it was saved.

5. Task 2 — load_to_snowflake

conn = snowflake.connector.connect(...)
cur = conn.cursor()
cur.execute(f"USE DATABASE {SNOWFLAKE_DB}")
cur.execute(f"USE SCHEMA {SNOWFLAKE_SCHEMA}")

for table, files in local_files.items():
    fq_table = f"{SNOWFLAKE_DB}.{SNOWFLAKE_SCHEMA}.{table}"
    for f in files:
        cur.execute(f"PUT file://{f} @%{fq_table}")
    cur.execute(f"COPY INTO {fq_table} FROM @%{fq_table} FILE_FORMAT=(TYPE=PARQUET) ON_ERROR='CONTINUE'")

Plain English:

  1. Open a connection to Snowflake.
  2. Tell Snowflake which database and schema to use.
  3. For every Parquet file, PUT uploads it into Snowflake's internal "table stage" (a temp staging area specific to the target table).
  4. COPY INTO reads all staged files and bulk-inserts them into the Snowflake table.
  5. ON_ERROR='CONTINUE' means: if one row is bad, skip it and keep going — don't crash the whole batch.

6. DAG schedule

with DAG(
    dag_id="minio_to_snowflake_banking",
    default_args=default_args,
    schedule_interval="*/1 * * * *",   # every 1 minute
    start_date=datetime(2025, 1, 1),
    catchup=False) as dag:
    task1 >> task2

*/1 * * * * is cron notation for "every minute." catchup=False means: if the DAG has been paused, don't try to run all the missed minutes when it starts again.

Script 5 — The dbt SCD2 DAG

docker/dags/scd_snapshots.py

What it does

This DAG runs once a day. It uses dbt to transform the raw CDC data into clean, analytics-ready tables. It has three steps, chained together:

  1. dbt_run_staging — clean up the raw data, rename columns, fix data types
  2. dbt_snapshot — take a historical snapshot (SCD Type 2) of customers and accounts so you can see how they changed over time
  3. dbt_run_marts — build the final business-friendly tables (dimensions and facts) for reporting
Analogy: The raw layer is a pile of every receipt ever printed. dbt is the accountant who sorts them into filing cabinets, labels them, and writes a neat summary every day.

What is SCD Type 2?

"Slowly Changing Dimension Type 2" means: when a customer changes their email, you don't overwrite the old one. Instead you:

  • Mark the old row with an end date ("this email was valid until 2026-04-24")
  • Insert a new row with the new email and a start date

This way you can always answer "what was this customer's email on any past date?" — critical for regulatory audits.

The code

1. Default settings

default_args = {
    "owner": "airflow",
    "depends_on_past": False,
    "retries": 1,
    "retry_delay": timedelta(minutes=1),
}

If a task fails, try once more a minute later before giving up.

2. The three tasks

dbt_run_staging = BashOperator(
    task_id="dbt_run_staging",
    bash_command="cd /opt/airflow/banking_dbt_v2 && dbt run --select staging --profiles-dir /home/airflow/.dbt",
)

dbt_snapshot = BashOperator(
    task_id="dbt_snapshot",
    bash_command="cd /opt/airflow/banking_dbt_v2 && dbt snapshot --profiles-dir /home/airflow/.dbt",
)

dbt_run_marts = BashOperator(
    task_id="dbt_run_marts",
    bash_command="cd /opt/airflow/banking_dbt_v2 && dbt run --select marts --profiles-dir /home/airflow/.dbt",
)

dbt_run_staging >> dbt_snapshot >> dbt_run_marts

Each task is a BashOperator — it runs a shell command inside the Airflow container. The commands change into the dbt project folder and invoke dbt.

The last line dbt_run_staging >> dbt_snapshot >> dbt_run_marts is the order of execution: staging runs first, then snapshot, then marts. If any step fails, the next ones are skipped.

3. Schedule

schedule_interval="@daily"

Runs automatically once every day at midnight UTC.

Script 6 — The Metabase Dashboard

docker-compose.yml (Metabase service) + dashboards built in the browser

What it does

Metabase is the final layer of the pipeline — the part where humans actually see the data. It connects directly to Snowflake, reads from the dbt marts, and lets you build charts and dashboards by clicking — no SQL required.

Analogy: Everything before this was a kitchen — chopping, cooking, plating. Metabase is the dining table. It takes the finished dishes (Snowflake marts) and presents them in a way managers and analysts can actually consume.

Why Metabase?

  • No-code dashboards — non-technical users (managers, auditors) can build their own charts.
  • Native Snowflake driver — no extra setup; it speaks Snowflake out of the box.
  • Self-hosted & open-source — no per-seat licence fees, fits the cost-efficient ethos of the rest of the stack.
  • Lives in the same Docker network — one docker compose up brings it online alongside Postgres, Kafka, MinIO, and Airflow.

How it runs

Unlike scripts 1–5, Metabase isn't a Python file you execute. It's a Docker service defined in docker-compose.yml:

metabase:
  image: metabase/metabase:latest
  container_name: metabase
  restart: always
  ports:
    - "3000:3000"
  environment:
    MB_DB_FILE: /metabase-data/metabase.db
    JAVA_TIMEZONE: UTC
  volumes:
    - metabase_data:/metabase-data

Once running, you open http://localhost:3000 in a browser, complete the first-run wizard, and add Snowflake as a data source using the same credentials the Airflow DAG uses.

What the dashboard shows

The current dashboard — "Banking Pipeline Overview" — is an observability dashboard. Its job is to confirm that data is flowing end-to-end through every layer of the pipeline. Two charts:

  1. Total Amount by Transaction Type (bar chart) — sum of amount from fact_transactions grouped by transaction_type. Confirms all three event types (deposit, transfer, withdrawal) are landing in Snowflake.
  2. Account Type by Balance (donut chart) — sum of balance from dim_accounts grouped by account_type. Confirms both account types are present and the SCD2 snapshot model is preserving state correctly.

What it tells us at a glance

  • All three transaction types are landing — Debezium isn't silently dropping any.
  • Net deposit flow is positive — deposits exceed withdrawals.
  • Both account types are present in dim_accounts with clean numeric balances totalling $1.6M — COPY INTO is preserving types properly.
  • Transactions and balances are queryable end-to-end, proving the full pipeline (Postgres → Debezium → Kafka → MinIO → Snowflake → dbt) is operating as designed.
Metabase closes the loop — it turns the warehouse into something a manager can see. Five scripts get the data there; Metabase is what makes the work visible.

What This Means for Zambian Banking Systems

This pipeline isn't just a technical exercise — it's modelled on the real operational gaps inside Zambia's banking sector. Currencies are in ZMW, customer names are drawn from Bemba, Nyanja, Tonga, Lozi and Tumbuka, and the email domains (zamtel.co.zm, unza.zm) reflect the local internet landscape. Every design choice was made with a Zambian commercial bank, microfinance institution, or mobile money operator in mind.

In one sentence: this is a low-cost, open-source blueprint that lets any Zambian financial institution move from end-of-day batch reporting to near real-time visibility over customer activity — the same capability that Tier-1 global banks rely on, built on tools a small Lusaka team can actually run.

1. Real-time fraud and AML monitoring

Today most Zambian banks reconcile transactions overnight. By the time a suspicious withdrawal is reviewed the next morning, the money is already gone — often through mobile money cash-out points. Because this pipeline streams every INSERT from the core banking database into Snowflake within roughly a minute, fraud and Anti-Money-Laundering (AML) rules can fire while the customer is still at the ATM or agent booth.

  • Detect rapid-fire withdrawals from the same account across multiple branches
  • Flag transactions that breach Bank of Zambia (BoZ) reporting thresholds in real time
  • Spot structuring patterns (many sub-threshold deposits) before the daily batch closes

2. Bank of Zambia regulatory reporting

BoZ requires commercial banks to file regular returns: liquidity, large-cash-transaction reports, suspicious-transaction reports (STRs), foreign-currency exposure, and prudential ratios. The SCD Type 2 snapshots in this pipeline mean the warehouse can answer "what did this customer's KYC profile look like on 31 March?" — a question regulators ask during audits and one most current systems struggle to answer without restoring backups.

3. Mobile money and agent banking integration

Zambia's financial system is dominated by mobile money (MTN MoMo, Airtel Money, Zamtel Kwacha) and agent banking. The Kafka layer in this pipeline is the natural integration point: every transaction from a USSD session, agent POS, or mobile app can be published to the same Kafka cluster and end up in the same Snowflake warehouse. This breaks the silos between bank-owned channels and telco-owned wallets — giving one unified customer view.

4. Financial inclusion analytics

Roughly 40% of Zambian adults remain unbanked. To extend credit and savings products into rural Northern, Luapula, and Western provinces, banks need to understand transaction behaviour at a granular level — average ticket size, deposit cadence, cash-in vs cash-out balance per district. The dbt marts (fct_transactions, dim_accounts) make these analyses one SQL query away, instead of a multi-week IT request.

5. Cost-appropriate technology for the local market

Most enterprise data platforms (Informatica, IBM DataStage, Oracle GoldenGate) are priced for Wall Street, not Cairo Road. This pipeline uses entirely open-source components — Postgres, Debezium, Kafka, MinIO, Airflow, dbt, Metabase — paired with Snowflake's pay-per-second billing. A Zambian bank can run the entire stack on a single mid-size server during pilot, then scale only the warehouse layer as volumes grow.

6. Beyond commercial banks

The same architecture applies — with minimal change — to:

  • Microfinance institutions (MFIs) — Bayport, FINCA, Madison, Vision Fund — for loan-portfolio monitoring and PAR (Portfolio at Risk) tracking
  • SACCOs and village banks — to centralise member savings data without buying enterprise software
  • Mobile money operators — to power churn analytics, cash-flow forecasting, and float optimisation
  • Insurance and pensions — NAPSA, ZSIC, and private insurers — for premium and claim event tracking
Bottom line for Zambia: this pipeline is a working reference architecture that proves modern, real-time data engineering is achievable on a Zambian budget. It localises the data (ZMW, Zambian names, local email domains), respects local regulatory needs (BoZ audit trails via SCD2), and integrates naturally with the mobile-money-first reality of the Zambian financial sector.
